# 团队项目管理系统 — 产品需求文档 (PRD)

> **文档状态**：修订版  
> **版本**：v1.2  
> **粒度层级**：需求 (Requirement)  
> **来源**：交互式 HTML 原型分析 + 需求迭代（过程中持续优化）  
> **变更说明**：移除评审中状态，增加测试中；去掉验收标准相关内容；去掉迭代（Sprint）概念；需求编号改为手动填写；开发人员分配优化，每人可拥有多个职能，仅多职能人员需分配时选择

## 1. 产品概述

### 1.1 产品愿景

打造一套以**需求（Requirement）为最小管理粒度**的团队项目管理系统，覆盖需求从提出、评审、开发、测试到上线的全生命周期。让每个需求可追踪、可度量、可追溯。

### 1.2 核心价值

| 维度 | 价值 |
|------|------|
| **细粒度管理** | 以「需求」而非「任务」为原子单位，颗粒度适中，既不过细（如 task）也不过粗（如 epic） |
| **全链路追踪** | 每个需求都有完整的生命周期状态流转，配合评论动态形成审计轨迹 |
| **数据驱动** | 通过统计卡片和报表，实时掌握团队交付节奏和瓶颈 |
| **多视图协作** | 列表视图满足管理需求，看板视图满足执行需求 |

### 1.3 核心概念定义

- **需求（Requirement）**：系统的核心实体，代表一个独立可交付的功能单元。粒度介于 Epic 和 Task 之间。
- **状态流**：待评审 → 开发中 → 测试中 → 已上线
- **优先级**：P0（紧急）→ P1（高）→ P2（中）→ P3（普通）

## 2. 目标用户与角色

| 角色 | 核心诉求 | 使用频率 |
|------|----------|----------|
| **产品经理 (PM)** | 创建需求、管理 Backlog、排优先级、查看交付进度 | 每天 |
| **研发工程师 (Dev)** | 查看已分配需求、认领开发任务、更新状态、参与评论、流转需求。按职能分为前端、后端、数据三类 | 每天 |

## 3. 用户故事地图

### Epic 1: 需求全生命周期管理

| Feature | User Story |
|---------|------------|
| **F1.1 创建需求** | 作为 PM，我希望能填写需求编号、标题、描述、优先级、开发人员等信息来创建需求，以便开发团队理解并实现 |
| **F1.2 需求列表查看** | 作为团队成员，我希望能按项目、状态、优先级等多维度筛选查看需求列表，以便快速找到我关心的需求 |
| **F1.3 需求详情查看** | 作为开发人员，我希望能查看需求的完整信息（描述、开发人员、时间计划、评论历史），以便准确理解需求 |
| **F1.4 需求开发完成** | 作为开发人员，我能在开发完成后将需求状态更新为「测试中」，测试通过后由 PM 确认上线 |
| **F1.5 需求编辑** | 作为 PM，我希望能更新需求信息（优先级、描述、负责人等），以便应对需求变更 |
| **F1.6 需求删除** | 作为 PM，我希望能废弃不再需要的需求（软删除），以便保持需求列表整洁 |

### Epic 2: 需求协作

| Feature | User Story |
|---------|------------|
| **F2.1 需求上线确认** | 作为 PM，我能在测试通过后将需求状态更新为「已上线」，以便确认交付完成 |
| **F2.2 评论动态** | 作为团队成员，我能在需求详情页发表评论，以便对需求进行讨论 |
| **F2.3 开发人员分配** | 作为 PM，我能为需求指派开发人员（前端/后端/数据），并多人协同开发。每人拥有预设的职能集，分配时仅多职能人员需选择具体职能 |
| **F2.4 @提及通知** | 作为团队成员，我能在评论中 @提及他人，以便通知相关同事关注 |

### Epic 3: 可视化交付管理

| Feature | User Story |
|---------|------------|
| **F3.1 统计卡片** | 作为 PM/Dev，我希望能在一屏内看到核心数据概览，以便快速掌握项目状况 |
| **F3.2 看板视图** | 作为开发，我希望能以看板方式查看需求在各状态下的分布（可按前端/后端/数据筛选），以便直观了解工作进度 |
| **F3.3 拖拽流转** | 作为开发，我希望能通过拖拽在看板上变更需求状态，以便快速更新进展 |
| **F3.4 最近投产** | 作为 PM，我希望能侧边栏快速查看最近需要投产的需求，以便把控交付节奏 |

## 4. 功能需求详述

### M1: 仪表盘 (Dashboard)

**位置**：侧边菜单第一项，作为系统首页

| 功能 | 说明 |
|------|------|
| **M1.1 统计卡片** | 展示 4 个核心指标：总需求数、进行中、待评审、本期已上线。每个卡片含环比变化 |
| **M1.2 最近投产需求** | 侧边栏底部展示最近需要投产的需求：需求名称、编号、开发人员、计划投产时间 |
| **M1.3 快捷入口** | 仪表盘区域可放置「快速新建需求」「查看待办」等快捷入口 |

**验收标准**：
- 统计卡片数据与需求列表数据保持实时一致
- 数字变化有微动效（+3 用绿色箭头，-2 用红色箭头）
- 最近投产需求展示即将到期的需求，按计划投产时间排序

---

### M2: 需求管理 (Requirement Management)

**位置**：侧边菜单第二项，核心功能模块

#### M2.1 需求列表

| 字段 | 类型 | 说明 |
|------|------|------|
| 需求编号 | String | 创建时手动填写，格式 `REQ-{YYYY}-{NNN}`，如 REQ-2025-001 |
| 需求标题 | String | 简明扼要的描述 |
| 所属项目 | String | 如支付中台、用户增长、数据平台 |
| 优先级 | Enum | P0/P1/P2/P3，不同颜色标识 |
| 状态 | Enum | 待评审/开发中/测试中/已上线 |
| 开发人员 | Array[Tag] | 按职能细分为前端/后端/数据，可多选。如「李浩(前端), 王婷(后端)」 |
| 创建时间 | Date | 需求创建日期 |
| 操作 | Action | 查看、编辑、删除 |

**筛选与搜索**：
- 关键字搜索（编号 + 标题）
- 项目下拉筛选
- 状态下拉筛选
- 优先级筛选
- 开发人员筛选（支持按职能类型筛选：前端/后端/数据）
- 筛选条件可组合，支持「重置」

**排序**：
- 默认按创建时间倒序
- 支持按优先级、状态、创建时间排序

**分页**：
- 每页默认 10 条，支持切换 10/20/50
- 显示总记录数

**验收标准**：
- 表格可横向滚动（当列数超出时）
- 行选中高亮效果
- 空数据时展示空状态引导
- 导出按钮支持 CSV 导出

#### M2.2 状态流转

```
               ┌──────────┐
               │ 待评审    │
               └────┬─────┘
                    │ PM 确认开发
               ┌────▼─────┐
               │ 开发中    │
               └────┬─────┘
                    │ 开发完成
               ┌────▼─────┐
               │ 测试中    │
               └────┬─────┘
                    │ PM 确认上线
               ┌────▼─────┐
               │ 已上线    │
               └──────────┘
```

**状态变更约束**：
- 不允许跨越状态变更（如从「待评审」直接到「测试中」）
- 状态变更需记录操作人、时间，写入审计日志
- 已上线状态不可回退
- PM 负责「待评审→开发中」「测试中→已上线」流转
- DEV 负责「开发中→测试中」流转

#### M2.3 创建需求表单

| 字段 | 必填 | 组件类型 | 说明 |
|------|------|----------|------|
| 需求编号 | ✅ | Input | 创建时手动填写，格式：REQ-2025-NNN |
| 需求标题 | ✅ | Input | 限制 100 字 |
| 需求描述 | ✅ | Textarea（3行） | Markdown 支持 |
| 所属项目 | ✅ | Select | 从项目列表动态加载 |
| 优先级 | ✅ | Select | P0/P1/P2/P3，默认 P2 |
| 开发人员 | ❌ | 人员多选 + 职能勾选 | 每人预设有职能集（前端/后端/数据），仅多职能人员需勾选具体职能 |
| 计划开发开始时间 | ❌ | DateTimePicker | |
| 计划开发结束时间 | ❌ | DateTimePicker | |
| 计划提测时间 | ❌ | DateTimePicker | |
| 计划测试周期（天） | ❌ | Number Input | |
| 计划投产时间 | ❌ | DateTimePicker | |

**验收标准**：
- 表单字段实时校验，提交时全量校验
- 必填字段有红色 `*` 标记
- 优先级有颜色标识辅助选择
- 开发人员选择时，每人基于预设职能集勾选，单一职能人员自动分配（无需勾选）
- 提交成功后自动跳转到需求列表并显示成功 Toast

#### M2.4 需求详情

**基本信息区**：
- 需求编号、所属项目
- 优先级标签（带颜色）、状态标签（带颜色）
- 创建人、创建时间

**需求描述区**：
- 完整描述文本（支持 Markdown 渲染）

**开发人员区**：
- 展示所有指派的开发人员列表，每人标注职能标签（前端/后端/数据）
- 每位开发人员显示头像、姓名、职能标签

**时间计划区**（一个时序时间线视图）：

| 里程碑 | 计划时间 | 实际时间 | 状态 |
|--------|----------|----------|------|
| 开发开始 | 2025-06-16 | 2025-06-16 | ✅ 按时 |
| 开发结束 | 2025-06-22 | - | ⏳ 进行中 |
| 提测 | 2025-06-23 | - | ⏳ 待开始 |
| 测试周期 | 3 天 | - | ⏳ 待开始 |
| 投产 | 2025-06-26 | - | ⏳ 待开始 |

**最新动态（评论时间线）**：
- 按时间倒序展示操作记录和评论
- 支持新增评论
- 系统自动记录状态变更（如「李浩 将状态从 待评审 变更为 开发中」）
- 评论支持 @提及

### M3: 项目看板 (Kanban)

**位置**：侧边菜单第三项

| 列名 | 对应状态 | 卡片内容 |
|------|----------|----------|
| 待评审 | pending_review | 需求标题、优先级、项目 |
| 开发中 | developing | 需求标题、优先级、开发人员及职能标签、时间计划 |
| 测试中 | testing | 需求标题、优先级、开发人员 |
| 已上线 | launched | 需求标题、优先级、投产时间 |

**交互**：
- 拖拽卡片跨列移动 → 触发状态变更
- 变更前需二次确认（如从「开发中」拖到「测试中」）
- 视觉反馈：拖拽时卡片跟随，释放后自动定位
- 每列显示当前卡片数量

**验收标准**：
- 拖拽流畅无卡顿
- 非法流转（如直接到「已上线」）弹窗阻止
- 移动端不支持拖拽，改为点击选择状态

## 5. 数据模型 / ERD

### 核心实体关系

```
Project ──────┐
              ├── Requirement ──── Comment
              │       │
              │  User ──┤ (创建人)
              │       │
              │  User ──┤ (开发人员, 多对多, 带职能角色)
              │       │
              │ StatusTransition (审计日志)
```

### 实体定义

#### Requirement

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| code | String | 需求编号，创建时手动填写，格式 REQ-2025-001 |
| title | String(200) | 需求标题 |
| description | Text | 需求描述，支持 Markdown |
| project_id | UUID FK | 所属项目 |
| priority | Enum | p0/p1/p2/p3 |
| status | Enum | pending_review / developing / testing / launched |
| creator_id | UUID FK | 创建人 |
| developers | JSON | 开发人员数组 [{user_id, roles: ["前端"|"后端"|"数据"]}]，每人可承担多个职能 |
| planned_dev_start | DateTime | 计划开发开始时间 |
| planned_dev_end | DateTime | 计划开发结束时间 |
| planned_test_submit | DateTime | 计划提测时间 |
| planned_test_days | Int | 计划测试周期（天） |
| planned_launch | DateTime | 计划投产时间 |
| actual_dev_start | DateTime | 实际开发开始时间 |
| actual_dev_end | DateTime | 实际开发结束时间 |
| actual_test_submit | DateTime | 实际提测时间 |
| actual_launch | DateTime | 实际投产时间 |
| tags | JSON | 标签数组 [string] |
| is_deleted | Boolean | 软删除标记 |
| created_at | DateTime | |
| updated_at | DateTime | |

#### Project

| 字段 | 类型 |
|------|------|

| lead_id | UUID FK |
| is_active | Boolean |

#### StatusTransition

| 字段 | 类型 |
|------|------|
| id | UUID |

| to_status | Enum |

## 6. 状态机定义

```typescript
const STATUS_FLOW = {
  pending_review: {
    label: '待评审',
    allowedTransitions: ['developing'],
    // developing = PM 确认开始开发
    requiredRole: ['pm'],
  },
  developing: {
    label: '开发中',
    allowedTransitions: ['testing'],
    // testing = 开发完成提交测试
    requiredRole: ['dev'],
    validation: { requireDeveloper: true },
  },
  testing: {
    label: '测试中',
    allowedTransitions: ['launched'],
    // launched = PM 确认上线
    requiredRole: ['pm'],
  },
  launched: {
    label: '已上线',
    allowedTransitions: [],
    requiredRole: ['pm', 'dev'],
  },
};
```

## 7. 非功能需求

### 7.1 性能

| 指标 | 目标值 |
|------|--------|
| 页面加载 | < 2s (1000 条需求) |
| API 响应 | < 200ms (95%) |
| 列表查询 | < 500ms (含筛选排序) |
| 并发用户 | 50 人同时使用 |

### 7.2 安全

- **RBAC**：基于角色的权限管理（PM / Dev）
- **审计日志**：所有状态变更、删除操作记录操作人、时间、变更内容
- **数据隔离**：用户只能看到自己有权限的项目
- **软删除**：需求删除不走物理删除

### 7.3 可用性

- 系统可用性 > 99.5%（每月非计划停机 < 3.6 小时）
- 操作有反馈（Toast 成功/失败提示）
- 关键操作（删除、状态变更）需二次确认

### 7.4 响应式

| 断点 | 布局 |
|------|------|
| ≥ 1024px | 完整左右布局，侧边栏展开 |
| 768px - 1023px | 侧边栏折叠为图标 |
| < 768px | 单列布局，底部导航 |

## 附录

### A. 术语表

| 术语 | 说明 |
|------|------|
| 需求 (Requirement) | 系统核心实体，代表一个可独立交付的功能单元 |
| P0 | 最高优先级，阻塞类需求 |
| P1 | 高优先级，核心功能 |
| P2 | 中等优先级，重要但非阻塞 |
| P3 | 普通优先级，优化类需求 |
| 开发人员职能 | 每个开发人员预设一个或多个职能（前端/后端/数据），分配需求时仅多职能人员需选择具体职能 |
| 测试中 | 开发完成后的测试阶段，DEV 提交测试，PM 确认上线 |
| 提测时间 | 开发完成后提交测试的时间节点 |
| 投产时间 | 需求正式发布上线的计划/实际时间 |
| RBAC | Role-Based Access Control，基于角色的权限控制 |
